⚙️ Update Workflow
Update Check
On startup (or at intervals), your app queries the GitHub Releases API (or your update server).
Compare the local version (e.g., v0.0.2) with the latest release tag.

Download Latest File
If a newer version exists, fetch the asset URL (e.g., BrightnessController.exe).
Download it to a same location where current program is (update_temp.exe).

Replace Old Program
Since a running executable cannot overwrite itself, use a helper updater script:
    The main app spawns a small updater process (batch file, PowerShell, or a tiny executable).
    The updater waits until the main app exits.
    It then replaces the old file with the new one.
    Finally, it restarts the updated program.

🔑 Key Points
    Don’t overwrite while running → use a helper updater.
    Rollback option → keep older versions accessible.


Background Download & Seamless Swap
    Download updates silently in the background while the app runs.
    Only prompt the user when the update is ready, minimizing disruption.

Atomic Replacement
    Use a temporary staging folder (update_temp) and only swap files once the download is verified.
    Prevents corruption if the update process is interrupted.

Rollback Mechanism
    Keep a backup of the old version (BrightnessController_old.exe).
    If the new version fails to launch, automatically revert to the previous one.

User Preferences
Allow users to choose:
    Silent updates (auto-install).
    Prompted updates (ask before replacing).
    Manual updates (disable auto-check).

UI Integration
    Add a small update notification in the app UI (e.g., “New version available: v0.0.3”).
    Include a changelog popup so users know what’s new.

Telemetry & Logging
    Log update attempts, successes, and failures.
    Helps diagnose issues and improve reliability.
